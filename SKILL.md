# 会议记录排版格式与行文风格指南

> 基于《集团领导调度理想之地营销工作会议记录》实际分析 + 国企/党政机关公文排版标准

---

## 〇、预处理流程（上传Word文档必做）

> 收到用户上传的 .docx 文件时，**必须先清除原格式再重新排版**，不得在原格式基础上修改。

### 0.1 处理流程

```
用户上传.docx文件
    ↓
① 读取原始文本内容（python-docx 提取纯文本）
    ↓
② 忽略原文档所有格式信息（字体/字号/加粗/颜色/行距/缩进/对齐方式）
    ↓
③ 根据文本结构识别层级：
   - 居中的短文本 → 文档总标题
   - "一、""二、"开头 → 一级章标题
   - "（一）""（二）"开头 → 二级节标题
   - "1、""2、"或"1.""2."开头 → 三级条标题
   - 其余 → 正文
   ↓
④ 按 §1.2 字体与字号层级表重新生成文档
   ↓
⑤ 应用 §1.3 段落间距规范（全部固定值28磅EXACTLY模式、space_before/after=0、首行缩进2字符宽度）
   ↓
⑥ 设置数字字体为 Times New Roman（§1.2 数字字体）
   ↓
⑦ 输出格式化 .docx
```

### 0.2 清除格式规则

| 操作 | 说明 |
|------|------|
| 清除字体 | 所有段落恢复默认，再按层级分别设为黑体/楷体/仿宋 |
| 清除字号 | 统一清除原字号，按层级设为22pt或16pt |
| 清除加粗 | 清除原加粗状态，仅三级条标题和字段前缀保留加粗 |
| 清除行距/段距 | 清除原倍数行距，全部设为固定值28磅 |
| 清除缩进 | 清除原缩进设置，统一按规范重设（总标题无缩进，其余2字符宽度） |
| 清除对齐方式 | 清除原对齐，总标题居中，其余左对齐 |
| 清除列表符号 | 原文档的项目符号（• - *等）全部转为正文文本 |

### 0.3 层级识别细则

识别文本层级时，按以下优先级判断（匹配即止）：

```
① 全文唯一居中的长文本 → 文档总标题
② 行首匹配"一、""二、"..."七、"等 → 一级章标题
③ 行首匹配"（一）""（二）""（1）"等 → 二级节标题
④ 行首匹配"1、""2、""1.""2."等 → 三级条标题
⑤ 以上均不匹配 → 正文
```

**注意**：原文档中可能有多级标题混用不同编号风格（如"1."与"1、"混用），统一按格式规范重新编号，保持一致性。

## 一、排版格式规范

### 1.1 页面布局

| 规格项 | 标准值 | 说明 |
|--------|--------|------|
| **纸张** | A4（8.27" × 11.69"） | 国际通用标准 |
| **上边距** | 2.54cm（1.0英寸） | 标准公文上边距 |
| **下边距** | 2.54cm（1.0英寸） | 对称一致 |
| **左右边距** | 3.18cm（1.25英寸） | 左侧装订留空 |
| **有效版心** | ~14.66cm宽 | 适合中文正文排布 |

### 1.2 字体与字号层级

| 层级 | 推荐字体 | 推荐字号 | 加粗 | 用途 |
|------|---------|---------|------|------|
| **文档总标题** | 方正公文小标宋 | 二号（22pt） | 否 | 文档最顶端的标题，居中排列 |
| **一级章标题** | **黑体** | **三号（16pt）** | 可加粗 | "一、""二、""三、"编号的章节标题 |
| **二级节标题** | **楷体** | **三号（16pt）** | **不加粗** | "（一）""（二）"编号的节标题 |
| **三级条标题** | **仿宋** | **三号（16pt）** | **加粗** | "1、""2、""3、"编号的条标题 |
| **正文** | **仿宋** | **三号（16pt）** | 否 | 会议记录的主体文字 |
| **行间距** | **全部固定值28磅** | — | — | 所有段落（含标题）统一固定值28磅行距，配合 **EXACTLY 模式** + **space_before=0 / space_after=0** 确保段落间与行间完全一致 |
| **段前间距** | **显式设为0磅** | — | — | 必须显式设置 Pt(0)，不能依赖默认值 |
| **段后间距** | **显式设为0磅** | — | — | 必须显式设置 Pt(0)，不能依赖默认值 |
| **首行缩进** | **2字符宽度** | — | — | **全部段落缩进（总标题除外）**：一级/二级/三级标题和正文均有首行缩进 **2个字符宽度（2个汉字宽度）**。使用 `w:firstLineChars="200"` XML 属性设置，而非 `Cm(0.75)` |
| **数字字体** | **Times New Roman** | — | — | 文档中所有阿拉伯数字统一使用 Times New Roman，与中文字体区分 |

#### 字号对照表

| 中文字号 | 对应磅值（pt） | 使用位置 |
|---------|-------------|---------|
| 二号 | 22pt | 文档总标题 |
| 三号 | 16pt | 一/二/三级标题 + 正文 |

#### 字体安装说明

| 字体 | 系统预装 | 获取方式 |
|------|---------|---------|
| 方正公文小标宋 | ❌ 非Windows预装 | 需下载安装包 |
| 黑体 | ✅ Windows系统自带 | 无需安装 |
| 楷体 | ✅ Windows系统自带 | 无需安装 |
| 仿宋 | ✅ Windows系统自带 | 无需安装 |

> **备选方案**：若无法获取"方正公文小标宋"，可用 **宋体（SimSun）** 替代，视觉效果接近。

### 1.4 段落间视觉层次

| 元素 | 格式处理 | 效果 |
|------|---------|------|
| 总标题 | 居中，无首行缩进 | 与正文形成排版差异 |
| 一级/二级/三级标题 | **均有首行缩进2字符宽度** | 所有内容统一缩进风格 |
| 行距统一 | 全部固定值28磅（EXACTLY模式）+ space_before=0 / space_after=0 | 全文段落间和行间距离完全一致 |
| 标题段独占一行 | 标题后紧跟正文段落（不合并） | 结构清晰 |
| 正文内字段强调 | 关键标签加粗（如"会议时间：""参会人员："） | 快速定位信息 |
| 数字字体 | 所有阿拉伯数字使用 Times New Roman | 与中文字体形成对比，数字清晰 |

## 二、行文风格指南

### 2.1 语言正式度

| 维度 | 标准 | 示例 |
|------|------|------|
| **整体基调** | **高度正式** | 使用"研判""锚定""攻坚""深化"等公文高频词汇 |
| **语气** | 客观陈述 + 指令性 | "需立即调整打法""需把提升来访量作为首要任务" |
| **人称** | 无第一/第二人称 | 全篇不使用"我""我们""你""您" |
| **标准度** | 严格遵循公文语体 | 句式完整，主谓宾齐全，无口语化表达 |

### 2.2 句式结构

| 句式类型 | 建议占比 | 典型用法 |
|---------|---------|---------|
| **陈述句**（告知信息） | ~30% | 会议基本信息、数据陈述、情况说明 |
| **祈使句**（提出要求） | ~55% | 董事长指示、工作部署、行动要求 |
| **判断句**（分析定性） | ~15% | 问题诊断、原因分析、效果评估 |

**句式要点：**

1. **祈使句主导**：以"需/要/需把/需通过/可借鉴/建议"等指令性动词开头，体现会议纪要的目标导向
2. **并列结构高频**：大量使用"、"串联并列动作，体现公文"一言多事"的高效表述习惯
3. **句子长度**：一般控制在 **30-60字/句**，核心议题等重要信息不超过80字

| 术语类别 | 示例 | 使用原则 |
|---------|------|---------|
| **行业术语** | 认筹锁客、推售策略、小阳春、来访转化、分销渠道 | 准确使用，不生造 |
| **营销术语** | 全域宣传、事件营销、垂类媒体、小V矩阵、圈层营销 | 使用行业通用说法 |
| **政企术语** | 党建联建、工会联动、党校实践教学 | 与官方用语一致 |
| **专有名词** | "五改四好""万小二" | 首次出现加引号标识 |

> **术语密度**：一般保持在 **0.3~0.4个术语/段** 为佳。过多则生涩难懂，过少则缺乏专业感。

### 2.4 文档逻辑结构

会议记录的标准逻辑链条：

```
会议基本信息（背景层）          → 时间、地点、人员、议题
    ↓
领导工作指示（战略层）           → 形势判断、核心要求、方向定调
    ↓
核心工作举措（执行层）           → 分维度、分条目的具体行动
    ↓
工作要求（保障层）               → 责任分工、时间节点、协同机制
```

**结构原则**：
1. **自上而下**：从背景到战略到执行到保障，层层递进
2. **先总后分**：先概述全局，再逐项展开
3. **问题→对策**：先分析问题/原因，再提出解决方案
4. **因果闭合**：每条举措应有明确的驱动因素或预期效果

### 2.6 分析逻辑参考

问题诊断、策略分析时，可参考以下思路（详见 `maoxuan-skill`）：

- **抓主要矛盾**：复杂局面中必有一个矛盾起主导作用，解决它就带动其他问题连带解决
- **重视一线信息**：没有调查就没有发言权，决策前先从一线获取信息
- **分清不同群体**：业主、拆迁户、教育局、街道办——利益不同，策略不同，不能一概而论
- **战略上藐视、战术上重视**：方向看准但每一步走稳，不急不躁
- **持久心态**：维稳、舆情的化解需要时间，不追求一蹴而就

以上思路融入分析但不贴标签，让结论自然成立。

## 三、中文公文AI味儿修正指南

本会话中用户多次纠正AI写作风格，以下为针对中文国企公文场景的特定要求。

### 3.1 必须避免的结构

> 完整的中文AI-ism词汇表和修正对照见 `references/chinese-gongwen-ai-isms.md`

| 问题 | 错误示例 | 正确写法 |
|------|---------|---------|
| **Em dash（—）** | 全程严守合规底线—实行专户专款 | 全程严守合规底线，实行专户专款 |
| **连字符（-）** | 列表用"－"或"1-2个月" | 列表用"•"，范围用"至"（"1至2个月"），正文中禁止出现"-"字符 |
| **空话堆砌开头** | 为坚决打赢全年20亿元销售目标攻坚战，稳固项目价格体系、坚守高端品质定位，抢抓...窗口期 | 直接写背景：江汉区华安里城市更新项目推出房票安置政策... |
| **三重并列（规则三）** | 快速扩大销售规模、加速资金回笼、平滑市场淡季波动 | 只用一句：拉动销售去化，对冲淡季波动 |
| **AI高频词** | 精准撬动、彰显国企担当、强力支撑、多方共赢、赋能 | 直接说做了什么，不要说\"彰显\"\"撬动\"\"赋能\" |
| **空泛结论** | 业务合规、流程闭环、风险可控 | 直接写结论和建议事项 |

### 3.2 用户偏好的结构风格

1. **先说要点，再展开。** 每个小节第一句话就是结论，随后是支撑细节。
2. **两种东西对比时，各自独立成节。** 不要揉在一起写，每个模式单独写\"分析\"和\"应对\"。
3. **段落短，每段只讲一个事。** 一段超过5行就拆。
4. **不要用\"本方案严格遵照...，建议...\"这类模板收尾。** 直接写结论和建议。

### 3.3 章节排布模板

适用于方案类公文（如房票实施方案）：

```
## 一、背景
（简明扼要，不堆砌空话）

## 二、两种模式

### 模式一：XX
#### 1. 模式分析
（政策约束、数据、条款）
#### 2. 应对措施
（每条措施配一个理由）

### 模式二：XX
#### 1. 模式分析
#### 2. 应对措施

## 三、实施计划
（时间表）

## 四、提请事项
```

### 敏感性分析表格式

适用场景：定价测算、成本分析、方案比选。要求完整列出所有变量，让读者看到各个因素如何叠加。

```
| 持有周期 | 变量A | 变量B | 综合折让 | 可接受价位 | 风险等级 |
|---------|------|------|---------|----------|---------|
| 3个月   | 5%   | 1%   | 6%      | 94折     | 低      |
| 6个月   | 5%   | 2%   | 7%      | 93折     | 低      |
| 1年     | 5%   | 4%   | 9%      | 91折     | 中      |
```

禁止做法：省略变量列、只给结论不给推导过程、用模糊描述代替数字。每个变量都要标注计算依据。```

## 四、常见问题与修正

### 3.1 高频错误对照表

| 常见错误 | 问题说明 | 正确做法 |
|---------|---------|---------|
| ❌ 使用西文字体 | Times New Roman 等西文衬线体汉字显示细瘦 | 中文正文用仿宋/宋体 |
| ❌ 一级标题不加粗 | 章标题与正文视觉无区分 | 黑体三号（可加粗） |
| ❌ 二级标题加粗 | 楷体加粗破坏手写亲和感 | 楷体不加粗 |
| ❌ 依赖样式继承字号 | 未显式设置字号，不同Word版本渲染不同 | 每级显式指定16pt |
| ❌ 行距1.25-1.5倍 | 倍数行距在中文文档中偏紧凑 | 固定值28磅 |
| ❌ 句子超80字 | 长句信息过载 | 拆分为2-3个短句 |
| ❌ 术语堆砌 | 同一段落出现3个以上专业术语 | 控制术语密度 |
| ❌ 段落间距不一致 | `line_spacing = Pt(28)` 未指定 EXACTLY 模式，或未清除 space_before/after | 同时设置 `line_spacing_rule=EXACTLY`、`line_spacing=Pt(28)`、`space_before=0`、`space_after=0` |
| ❌ 缩进量错误 | 使用 `Cm(0.75)` 仅约21pt，不到三号字2汉字宽度 | 使用 `w:firstLineChars="200"`（2字符宽度） |
| ❌ 标题无首行缩进 | 标题从页边距起排，视觉突兀 | 除总标题外，所有标题统一首行缩进2字符宽度 |
| ❌ 不清除原格式直接修改 | 在原有格式基础上叠加新格式，导致字体/行距冲突 | 读取纯文本后，从空白文档重新生成 |
| ❌ 数字字体混用 | 数字跟随中文仿宋字体，比例不适 | 所有阿拉伯数字设为 Times New Roman |

### 3.2 快速检查清单

生成或审阅会议记录时，逐项确认：

#### 排版格式

- [ ] **预处理：已清除原文档所有格式**（字体/字号/加粗/行距/缩进/对齐）
- [ ] 总标题：方正公文小标宋（或宋体）/ 22pt / 居中 / **无首行缩进**
- [ ] 一级标题：黑体 / 16pt / **首行缩进2字符宽度**
- [ ] 二级标题：楷体 / 16pt / 不加粗 / **首行缩进2字符宽度**
- [ ] 三级标题：仿宋加粗 / 16pt / **首行缩进2字符宽度**
- [ ] 正文：仿宋 / 16pt / 首行缩进2字符宽度
- [ ] 行距：**全部段落**固定值28磅（含标题），**必须使用 EXACTLY 模式** + space_before=0 + space_after=0
- [ ] 数字字体：所有阿拉伯数字使用 **Times New Roman**
- [ ] 落款：右对齐 + 仿宋16pt正文同款

#### 内容规范

- [ ] **删减清单**：删重复词、删客套话、删能对任何人说的空话、把客套换成具体细节
- [ ] 正文中无 **em dash（—）**，改用逗号或句号
- [ ] 正文中无 **连字符（-）**，列表用"•"，范围用"至"
- [ ] 无AI空话堆砌开头（"为坚决打赢..."之类），直接写背景
- [ ] 无高频AI词（精准撬动、彰显、赋能、强力支撑、多方共赢等）
- [ ] 无三重并列（"快速扩大、加速、平滑"这类），精简为一句
- [ ] 标题与正文：分段落，不合并
- [ ] 段落内字段强调：使用加粗而非字体字号变化
- [ ] 样式使用：建立Word样式模板（Style Set）后一键套用
- [ ] 编号一致性：一→（一）→1→（1）层级递进正确
- [ ] 句子长度：最长不超过80字
- [ ] 祈使句驱动：指令明确，动作导向
- [ ] 术语密度：0.3~0.4个术语/段
- [ ] 逻辑链条：信息→指示→举措→要求 完整

### 3.3 Word样式模板设置建议

在Word中建立样式模板（Style Set），将内置样式预设为：

skills 目录下附带脚本 `scripts/generate-from-markdown.py`，用于将Markdown格式的分析报告/会议记录直接转换为按公文规范排版的Word文档。用法：

**Markdown源文件规范要求：**

| Markdown元素 | 排版对应 | 说明 |
|-------------|---------|------|
| `# 标题` | 文档总标题（宋体22pt居中） | 全文第一个`#`视为总标题，后续`#`降级处理 |
| `## 标题` | H1级（黑体16pt加粗） | 对应"一、""二、"编号的一级标题 |
| `### 标题` | H2级（楷体16pt不加粗） | 对应"（一）""（二）"编号的二级标题 |
| `#### 标题` | H3级（仿宋16pt加粗） | 对应"1、""2、"编号的三级标题 |
| `\| 表头 \| 表头 \|` | 居中表格 | 首行黑体12pt加粗，数据行仿宋12pt |
| `> 引用` | 楷体16pt加粗斜体 | 适合关键结论、核心洞察等高亮内容 |
| `---` | 分隔线 | 章节之间的视觉分隔 |
| `**加粗**` | 仿宋16pt加粗 | 正文内字段强调 |

**支持的处理特性：**
- 首行`：`前缀自动识别为"标签：内容"格式（标签加粗、内容正常）
- **自动落款右对齐**：识别文档末尾的落款行（公司名/集团/局/部 + 日期），自动设为右对齐、仿宋16pt，与正文一致
- `---`分隔符不渲染在落款前（落款前若有多余横线，建议在MD中删除）
- 正文内的`**加粗文本**`标记自动转换为加粗
- 表格自动识别（连续`|`行组成表格，含`|---|`分隔线识别）
- 所有阿拉伯数字自动设为Times New Roman

### 4.2 python-docx 生成标准化会议记录框架（参考代码）

```python
from docx import Document
from docx.shared import Pt, Cm, Emu
from docx.enum.text import WD_ALIGN_PARAGRAPH, WD_LINE_SPACING, WD_LINE_SPACING
from docx.oxml.ns import qn

doc = Document()

# --- 页面设置 ---
section = doc.sections[0]
section.page_width = Cm(21)
section.page_height = Cm(29.7)
section.top_margin = Cm(2.54)
section.bottom_margin = Cm(2.54)
section.left_margin = Cm(3.18)
section.right_margin = Cm(3.18)

# --- 样式设置函数 ---
def set_style(run, font_name, size_pt, bold=False):
    run.font.name = font_name
    run.font.size = Pt(size_pt)
    run.font.bold = bold
    run.font.element.rPr.rFonts.set(qn('w:eastAsia'), font_name)
    if font_name != 'Times New Roman':
        # 保留中文字体，确保数字后续单独设置Times New Roman
        pass

# --- 设置段落格式（通用）：行距固定值28磅EXACTLY + space_before/after归零 + 首行缩进2字符宽度 ---
def set_paragraph_format(p, indent=True):
    pf = p.paragraph_format
    pf.line_spacing_rule = WD_LINE_SPACING.EXACTLY  # 固定值模式
    pf.line_spacing = Pt(28)                          # 固定值28磅
    pf.space_before = Pt(0)                           # 段前0磅（必须显式）
    pf.space_after = Pt(0)                            # 段后0磅（必须显式）
    if indent:
        # 使用 XML 设置首行缩进2字符宽度（w:firstLineChars）
        # 三号字16pt下，2字符 = 2个汉字字符宽度
        ind = p._element.get_or_add_pPr().find(qn('w:ind'))
        if ind is None:
            from lxml import etree
            ind = etree.SubElement(p._element.get_or_add_pPr(), qn('w:ind'))
        ind.set(qn('w:firstLineChars'), '200')  # 2字符宽度
        # python-docx 的 first_line_indent 也会设置，但 XML 层级更高

# --- 总标题 ---
p = doc.add_paragraph()
p.alignment = WD_ALIGN_PARAGRAPH.CENTER
run = p.add_run('会议记录标题')
set_style(run, '方正公文小标宋', 22)
# 备选：set_style(run, '宋体', 22)

# 空段
doc.add_paragraph()

# --- 一级标题 ---
def add_h1(text):
    p = doc.add_paragraph()
    set_paragraph_format(p, indent=True)  # 首行缩进2字符宽度 + 固定值28磅EXACTLY + space_before/after=0
    run = p.add_run(text)
    set_style(run, '黑体', 16, bold=True)

# --- 二级标题 ---
def add_h2(text):
    p = doc.add_paragraph()
    set_paragraph_format(p, indent=True)  # 首行缩进2字符宽度 + 固定值28磅EXACTLY + space_before/after=0
    run = p.add_run(text)
    set_style(run, '楷体', 16, bold=False)

# --- 三级标题 ---
def add_h3(text):
    p = doc.add_paragraph()
    set_paragraph_format(p, indent=True)  # 首行缩进2字符宽度 + 固定值28磅EXACTLY + space_before/after=0
    run = p.add_run(text)
    set_style(run, '仿宋', 16, bold=True)

# --- 正文（带可选加粗前缀） ---
def add_body(text, bold_prefix=None):
    p = doc.add_paragraph()
    set_paragraph_format(p, indent=True)  # 首行缩进2字符宽度 + 固定值28磅EXACTLY + space_before/after=0
    if bold_prefix:
        run = p.add_run(bold_prefix)
        set_style(run, '仿宋', 16, bold=True)
    run = p.add_run(text)
    set_style(run, '仿宋', 16)

# --- 设置数字字体为 Times New Roman ---
# ⚠️ 坑：p.add_run() 不能用来拆分已有的run，因为会同时修改 p.runs 迭代器。
#     必须使用 lxml XML 级操作：删除原 <w:r> 元素，在原地插入多个 <w:r> 子元素，
#     每个子元素包含正确的 <w:rPr>/<w:rFonts> 字体设置。
def set_number_font():
    """
    将文档中所有阿拉伯数字设为 Times New Roman。
    使用 lxml XML 级操作，避免 python-docx 的迭代器冲突。
    """
    from lxml import etree
    import re
    for p in doc.paragraphs:
        for run in list(p.runs):
            text = run.text
            if not text or not re.search(r'\d', text):
                continue
            parts = re.split(r'(\d+)', text)
            if len(parts) <= 1:
                continue
            orig_font = run.font.name
            orig_size = run.font.size
            orig_bold = run.font.bold
            rpr_elem = run._element.find(qn('w:rPr'))
            east_asia = None
            if rpr_elem is not None:
                rfonts_elem = rpr_elem.find(qn('w:rFonts'))
                if rfonts_elem is not None:
                    east_asia = rfonts_elem.get(qn('w:eastAsia'))
            parent = run._element.getparent()
            idx = list(parent).index(run._element)
            parent.remove(run._element)
            new_elements = []
            for part in parts:
                new_elem = etree.SubElement(parent, qn('w:r'))
                rPr_elem = etree.SubElement(new_elem, qn('w:rPr'))
                rFonts_elem = etree.SubElement(rPr_elem, qn('w:rFonts'))
                if re.fullmatch(r'\d+', part):
                    rFonts_elem.set(qn('w:ascii'), 'Times New Roman')
                    rFonts_elem.set(qn('w:hAnsi'), 'Times New Roman')
                    rFonts_elem.set(qn('w:eastAsia'), 'Times New Roman')
                else:
                    rFonts_elem.set(qn('w:ascii'), orig_font or '仿宋')
                    rFonts_elem.set(qn('w:hAnsi'), orig_font or '仿宋')
                    rFonts_elem.set(qn('w:eastAsia'), east_asia or orig_font or '仿宋')
                if orig_size:
                    sz = etree.SubElement(rPr_elem, qn('w:sz'))
                    sz.set(qn('w:val'), str(int(orig_size.pt * 2)))
                    szCs = etree.SubElement(rPr_elem, qn('w:szCs'))
                    szCs.set(qn('w:val'), str(int(orig_size.pt * 2)))
                if orig_bold:
                    etree.SubElement(rPr_elem, qn('w:b'))
                t = etree.SubElement(new_elem, qn('w:t'))
                t.text = part
                t.set('{http://www.w3.org/XML/1998/namespace}space', 'preserve')
                new_elements.append(new_elem)
            for ne in new_elements:
                parent.remove(ne)
            for j, ne in enumerate(new_elements):
                parent.insert(idx + j, ne)
```

> 完整的文档生成脚本建议保存为单独的脚本文件

### 4.3 数字字体设置避坑

设置数字字体为 Times New Roman 时，**不能使用 `p.add_run()` 拆分已有 run**（会导致迭代器冲突、文字重复），必须使用 lxml XML 级操作。详见 `references/docx-number-font-pitfall.md`。

## 五、融合式文档修订：在保留原文骨架的基础上融入新理念

当用户要求"在V1.0基础上融入新的核心理念，不要另起炉灶"时，采用**融合式修订**方法，而非标签式拼接。

### 5.1 核心原则：找内在呼应，不贴外部标签

错误的做法（标签式拼接）：
```
原文：巴黎时装周以传承与极致为核心……
融入：增加"人物传承"视角——谢东升老师13年读书班……
```
问题：两套语言系统强行拼接，读起来突兀，原文的"传承与极致"被稀释。

正确的做法（内在呼应）：
```
原文：巴黎时装周以传承与极致为核心…… 
融入：这种"传承"的内核，与理想之地正在培育的在地人物深度同频。
谢东升老师13年如一日的读书班……他们的坚持本质上与巴黎手工坊的
精神是同一种东西——人的热爱与时间的沉淀，才是最高级的奢侈。
```
关键：**先完全理解原文的内部逻辑，找到原文概念与新理念之间的共通基因，然后用原文自己的语言体系来表达这种共鸣。**

### 5.2 四步修订流程

```
① 沉浸阅读原文
   逐章通读，理解每章自己的逻辑体系和核心论点。
   不要跳过——只有在完全理解原文后才能找到正确的融入点。

② 为每章找到"共振点"
   问自己：这一章的核心理念，和新理念之间的共同基因是什么？
   例：巴黎"传承" → 谢东升13年读书班的坚持 = 共同基因是"时间的沉淀"
   例：伦敦"创新孵化" → 培养社区主理人 = 共同基因是"让人成长"

③ 用原文的语言体系表达融合
   先重现原文章节的核心论点（保持完整性），
   然后用"这种XX的内核，与XX同频""这一理念与XX的内在逻辑一致"等过渡句自然引入，
   最后用原文风格的词汇收束（"让传承有了可触摸的载体""让创意有了可持续的土壤"）。

④ 逐章验证
   检查每个被修改的段落：去掉融合标记后，原文逻辑是否依然完整成立？
   如果去掉后原文断裂 → 融合过深，需要调整
   如果去掉后原文完好 + 加上后自然延伸 → 融合成功
```

### 5.3 融入标记规范

在文档中标注每一处融入，方便审阅者识别：

| 标记 | 用法 | 示例 |
|------|------|------|
| `【理念融合】` | 融合后的内容 | `【理念融合】这传承的内核，与XX深度同频……` |
| 方括号标粗 | 每章首处标记 | `三、价值主张融合【理念融合】` |

标记不应打断阅读流畅性——融入后的段落应能独立成文，标记仅为审阅辅助。

### 5.4 常见陷阱

1. **标签式拼接**：用一套完全不同的语言体系插入原文，导致原文断裂。
   检查方法：朗读修改后的段落——如果听起来像"先说了A，突然跳到B"，就是标签式拼接。

2. **过度覆盖**：修改幅度超过40%的原文，导致V2.0变成新文档而非升级。
   原则：V2.0应保留V1.0至少60%的原文，融入内容占比不超过40%。

3. **丢失原文精华**：为了融入新理念而删掉原文有价值的论点。
   原则：保留原文每个章节的核心论点完整，融入是"加"不是"改"。

### 5.5 验证清单

- [ ] 原文每个章节的核心论点完整保留
- [ ] 融入点使用原文自己的语言体系表达
- [ ] 去掉融入标记后，原文逻辑链不断裂
- [ ] 融入内容占比不超过40%
- [ ] 每个融入点与所在章节有明显的内在逻辑关联（而非强行关联）
- [ ] 融入后的段落读起来"本来就是这样的"而非"被塞了东西"

## 七、多源文档→宣讲稿生成工作流

当用户上传多个源文件（活动方案PDF、优秀示例DOCX、初稿DOCX）并要求生成宣讲稿时，采用以下工作流：

### 7.1 四步法

```
① 读取并分析所有源文件
   → PDF用pymupdf, DOCX用python-docx, 注意安全拦截需写脚本文件到/tmp/
② 从优秀示例中提取叙事技法（场景化开场/冲突-反抗/人物故事锚点/今昔对照/情感高潮金句）
③ 按宣讲稿结构框架撰写正文（7部分：开场→历史→精神→当代→人物→升华→结语）
④ 多格式同步输出：Word（本skill排版）+ HTML演示文稿（next-slide skill）
```

详见 `references/speech-writing-workflow.md`，包含：多源文档读取方法、叙事技法提取表（5种经典技法）、宣讲稿结构框架、活动主题嵌入规则、多格式同步输出流程、常见陷阱对照表。

### 7.2 参考案例

配套实战案例见 `references/speech-writing-case-study-105.md`（建党105周年"跨越百年的初心"宣讲稿生成案例），包含：
- 四步升格策略决策树（项目叙事→政治主题的平滑迁移实战记录）
- 7段结构与5种叙事技法的映射表（每段字数、技法编号、升格要点）
- 5篇优秀范文→本案例的技法复用对照（从《脊梁》《大地母亲》等提取了哪些元素）
- 活动主题嵌入前后的原文逐段对比（25%改动、故事层0改动）
- 输出质量验证清单（7项全部通过）

该案例可作为同类任务的质量对标基准：输出稿件应至少在5项验证项上达到同等水平。

### 7.3 与 next-slide skill 的协同

当用户同时要求 Word + HTML 演示文稿时：
1. **Word**：按本skill的公文排版规范生成（宋体22pt标题、仿宋16pt正文、固定值28磅行距、首行缩进2字符、数字字体TNR）
2. **HTML**：由 next-slide 的 CSS+JS 框架生成，二者共用同一份 `paragraphs[]` 内容源
3. **保存路径**：Word → `01-报告/`，HTML → `04-策划方案/`

## 八、文件命名规范

> 用户明确要求：所有生成的文件必须遵循统一的命名体系，便于归档和检索。

### 8.1 核心规则

```
YYYYMMDD_项目_内容描述_V版本.扩展名
```

| 段位 | 要求 | 示例 |
|------|------|------|
| **YYYYMMDD** | 8位日期，定稿/生成日期 | `20260523` |
| **项目** | 短项目名，2-6字 | `理想之地` / `恒江雅筑` |
| **内容描述** | 具体文档类型，3-8字 | `房票安置实施方案` / `舆情应对实施方案` |
| **V版本** | 可选，大版本用 V1.0/V2.0 | `V4.0` |
| **扩展名** | `.docx` / `.md` / `.pdf` | `.docx` |

### 8.2 完整示例

```
20260523_理想之地_房票安置实施方案_V4.0.docx
20260523_恒江雅筑_舆情应对实施方案_V2.0.docx
20260523_恒江雅筑_统一解释话术手册_V2.0.docx
```

### 8.3 归档目录映射

输出/ 目录按三叉戟模式分流，目录编号连续、按用途分类：

| 文档类型 | 流转去向 | 说明 |
|---------|---------|------|
| 最终成果（报告/方案/分析） | → 源文件/ 对应分类目录 | 稳定后归档，删除输出中对应文件 |
| 经验模式（方法/规范/洞察） | → wiki/ 对应页面 | 提炼核心内容创建wiki页，原始文件若已完成则归档到源文件后删除 |
| 运行产物（脚本/配置/缓存） | 留在 输出/ | 长期驻留，不作为最终成果迁移 |
| 一次性记录、冗余副本 | → 删除 | 不归档不沉淀 |

输出/ 目录按用途编号，不按文档类型稀疏排列：

```
输出/
├─ 01-脚本/   — 数据抓取脚本、自动化工具（长期驻留）
├─ 02-舆情/   — 舆情监控系统（脚本+配置+数据，长期驻留）
├─ 03-数据/   — 运行时数据缓存（长期驻留）
```

命名规则：连续编号 + 短名称，每类只做一件事。无内容时目录删除不保留空壳。|

### 8.4 禁止的做法

| ❌ 错误 | 正确做法 |
|---------|---------|
| 无日期的文件名（如 `方案.docx`） | 加日期前缀 |
| 日期位置不统一（如 `方案20260523.docx`） | 日期固定在最前 |
| 中英文混用无规律（如 `planV2_final.docx`） | 统一用中文描述 |
| MD和docx不同名 | 保持同名，仅扩展名不同 |
| 文件名含括号版本（如 `(1)(4)(1)`） | 改用 V1.0/V2.0 标准化版本号 |

### 8.5 多版本迭代时的处理

| 场景 | 做法 |
|------|------|
| 生成初稿 | `V1.0` |
| 用户反馈修改 | `V2.0`，输出目录中**删除旧版文件**（只保留最新版本） |
| 最终定稿 | 去掉版本号或保留 V_Final |

> 注意：源文件目录中的历史版本（如 V3.0）应保留不删，版本管理只在 `输出/` 目录中按最新版本保留。

### 8.6 归档后清理

输出/ 是过渡暂存层，非永久归档。所有产出物按四叉戟模式分流：

① 最终成果 → 归档至 源文件/ 对应分类目录（设计/工程/成本/营销/运营/汇报等）
② 经验模式 → 沉淀至 wiki/ 对应页面（entities/ concepts/ comparisons/ queries）
③ 运行产物 → 留在 输出/（脚本、数据库、日志、运行中报告等持续产出）
④ 其余 → 清理删除

简化规则：
```
产出稳定 → 判断归属 → ①→源文件/ ②→wiki/ ③→留输出/ ④→删除
                          ↑ 只删已归档的，不删进行中的
```

操作铁律：每次分流后执行 /new 新会话，避免敏感数据交叉。

### 8.7 三步分流工作流

所有产出物按以下三阶段流转：

```
输入阶段     →   加工阶段     →   分流阶段
源文件/读取   →   输出/加工     →   ① 最终成果 → 源文件/
新资料输入         wiki同步           ② 经验模式 → wiki/
                                     ③ 运行产物 → 留输出/
                                     ④ 其余 → 删除
```

Wiki 在加工阶段末尾更新（50字起，写结论+关键数据），不等分流阶段。

目录编号规则：连续不跳号（01/02/03...），按用途命名。无内容时目录删除不留空壳。

### 8.8 报告追加模式

定期报告（舆情日报、销售日报、竞品周报、维稳简报）采用**追加模式**：每次新数据在页面末尾追加，**不覆盖历史**。这样形成逐日/逐周的趋势记录，支持回溯对比。

```
页面初始：
  [最新摘要/趋势总览]
  [历史日志]
    2026-05-22 完整详情
    2026-05-17 完整详情

追加后：
  [最新摘要/趋势总览] ← 自动更新
  [历史日志]
    2026-05-23 完整详情 ← 追加在此
    2026-05-22 完整详情
    2026-05-17 完整详情
```

追加模式适用于 Notion 页面和本地报告文件。Notion 写入时用 append 方式追加 blocks，不删除已有内容。
